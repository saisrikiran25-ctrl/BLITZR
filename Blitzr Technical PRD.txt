Technical Product Requirements Document (PRD): BLITZR
Version: 1.0
Project Code: BLITZR-PRIME
Status: Final Draft
Target: Engineering, DevOps, and Product Teams
1. Executive Summary
BLITZR is a high-concurrency campus social-equity exchange. It operates as a dual-market system: a Personal IPO Market (Equity) and a Prop Market (Derivatives). The system must handle high-velocity trading, real-time price updates, and anonymous social interactions while ensuring strict data integrity and preventing race conditions in a shared liquidity environment.
2. System Architecture & Tech Stack
2.1 Backend Architecture
* Primary Language: Node.js (TypeScript) or Go (for high-performance concurrency).
* Framework: NestJS (if Node) for modularity or Gin (if Go).
* Database: PostgreSQL (Relational integrity is non-negotiable for financial transactions).
* Caching/Real-time: Redis for Pub/Sub (Price updates) and caching high-frequency ticker data.
* WebSockets: Socket.io or Gorilla WebSockets for real-time chart wiggling and ticker tapes.
2.2 Frontend Architecture
* Framework: React Native (iOS/Android) for a unified codebase with native performance.
* State Management: Redux Toolkit or Zustand (optimized for high-frequency state updates).
* Charting: Highcharts or Lightweight Charts (TradingView) for performant line charts on mobile.
2.3 Infrastructure
* Cloud: AWS or GCP.
* Serverless: AWS Lambda for scheduled "Market Maker" bot scripts.
* Security: Cloudflare for DDoS protection and WAF.

4.1 Personal IPO Market (The "Wall Street" Tab)
            * User IPO Opt-in: A POST request that initializes a ticker_id, sets initial supply to 1 (held by creator), and calculates the starting price.
            * Real-time Ticker: A WebSocket stream delivering the current    and 24h % change.
            * Trade Execution: Atomic "Buy" and "Sell" operations.
            * Portfolio Tracking: Real-time calculation of Net Worth:   .
4.2 Prop Market (The "Arena" Tab)
               * Event Lifecycle: 1. Creation: Admin or User (with fee) defines event_name, options (Yes/No), and expiry_timestamp.
2. Active Trading: Users place Chips on options.
3. Resolution: An oracle (Admin or Referee) triggers the settlement.
               * Payout Logic: Pari-mutuel betting or Fixed-odds (Bonding curve based). Pari-mutuel is recommended for simplicity: Payout = (Total Pool - House Cut) * (User Bet / Winning Side Total Pool).
4.3 The Rumor Feed (Social Layer)
                  * Anonymity Layer: Every post generates a temporary ghost_id to prevent tracking.
                  * Ticker Tagging: Regex parser to identify $TICKER symbols and create deep-links to the Trading Floor.
                  * Moderation Engine: Integration with an NLP service (e.g., Perspective API or custom LLM) to block prohibited content before it hits the DB.

5.2 The "Atomic Race Condition" Fix
To prevent the "Priya Slap" scenario where 150 people buy at once:
                  * Database Transaction Isolation: Use SERIALIZABLE isolation level or SELECT ... FOR UPDATE on the tickers row.

6. Real-time Engine & Bot Logic
6.1 Market Maker Bot (Ghost Town Defense)
                     * Trigger: Cron job every 120 seconds.
                     * Logic: 1. Select 10 random active tickers with low volume.
2. Execute a random Buy or Sell of 1-3 shares.
3. This wiggles the line charts, creating visual "activity" and psychological incentive for users to investigate the ticker.
                     * Sunset Policy: Once a ticker hits 50 actual human trades per hour, the bot disengages.
6.2 Chart Data Aggregation
                        * OHLC Data: The system must aggregate trades into 1m, 5m, 1h, and 1d buckets for candle/line chart rendering.
                        * Optimization: Store historical ticks in a time-series optimized table (e.g., TimescaleDB) to prevent slow reads as the DB grows.
7. Security & Risk Management
7.1 The "Panic Button" Logic (Delisting)
                        1. User triggers POST /api/v1/ipo/delist.
                        2. System sets ticker.status = 'FROZEN'.
                        3. Refund Phase: A background worker iterates through all holdings for that ticker.
                        4. Total Creds equal to the current_price * shares_held are credited back to shareholders.
                        5. Ticker record is soft-deleted.
7.2 Anti-Fraud & Sybil Attacks
                        * Verification: Users must verify via a college-issued email (.edu or specific domain).
                        * Rate Limiting: Maximum 5 trades per minute per user to prevent high-frequency botting from exhausting the liquidity curve.
                        * Withdrawal/Transfer Lock: Since coins are fake, there is no withdrawal, but "Cred Transfers" between users must be logged to detect "Real Money" black market trading.
8. UI/UX Technical Specifications
8.1 Visual Performance
                        * Skeleton Screens: Used for the Trading Floor while WebSocket connections initialize.
                        * Haptic Feedback: Native vibration on trade success or "Nuke" activation.
                        * Color Palette:
                        * Bullish: #00C805 (Neon Green)
                        * Bearish: #FF3B30 (Alert Red)
                        * Background: #0A0A0A (OLED Black for depth)
8.2 Component Hierarchy
                        * MarketProvider: Global context for real-time prices.
                        * TradingTerminal: Modal for executing Buy/Sell orders with a "Review" step to show the impact of the bonding curve.
                        * EventCard: Dynamic component for the Arena that shows a countdown timer and live odds shifting.
9. Scalability & Deployment
9.1 Horizontal Scaling
                        * The API layer must be stateless to allow auto-scaling behind a Load Balancer.
                        * WebSocket connections must be handled by a dedicated service (e.g., AWS AppSync or a managed Redis Pub/Sub cluster) to maintain state across multiple server instances.
9.2 CI/CD Pipeline
                        * Staging Environment: A "Beta" college environment for testing new bonding curve constants.
                        * Automated Testing: Unit tests for the bonding curve math (Floating point precision check) and integration tests for atomic transactions.
10. Success Metrics (KPIs)
                        * T+0 Liquidity: Time taken for a trade to reflect in the global supply (<100ms).
                        * D1 Retention: Percentage of users who return to check their "Prop Bets" or "Dividend" earnings.
                        * Coin Velocity: Total volume of Chips traded vs. global Chip supply (measures economic health).
